# 🔧 Kubernetes Cluster on Ubuntu 24.04 — KVM + Ansible + Manifests

Automated deployment of a 3-node Kubernetes cluster on KVM/libvirt VMs using kubeadm and Ansible, including the full migration of a Docker Compose stack to native Kubernetes manifests.

## 📋 Requirements

| Requirement                         | Description                                                          |
| ----------------------------------- | -------------------------------------------------------------------- |
| KVM/libvirt                         | Hypervisor installed and configured                                  |

## 🚀 Getting Started

### Step 1 — Deploy VMs and prepare the environment

The `deploy-k8s-lab.sh` script automatically:

- Creates 3 KVM VMs (k8smaster1, k8sworker1, k8sworker2) with COW disks based on the cloud image
- Injects hostname, `/etc/hosts` and SSH key via cloud-init
- Reserves static IPs in libvirt's dnsmasq by MAC address
- Queries the latest available Kubernetes version from the official repository
- Generates `vars/variables.yml` from the `variables.yml.j2` template using `envsubst`
- Generates the Ansible inventory `inventory_hosts.yml`

```bash
chmod +x deploy-k8s-lab.sh
./deploy-k8s-lab.sh
```

### Step 2 — Run the playbook

```bash
ansible-playbook -i inventory_hosts.yml install_kubernetes.yml
```

### Step 3 — Copy the kubeconfig to the host

The playbook automatically copies the kubeconfig to the host if `copy_kubeconfig_to_host: true` is set in `vars/variables.yml`. To do it manually:

```bash
scp ubuntu@<IP_MASTER>:/home/kubernetesop/.kube/config ~/.kube/config
```

### Step 4 — Deploy the WordPress stack

```bash
kubectl apply -f ../manifests/wordpress/
```

### Step 5 — Add entries to /etc/hosts

```
192.168.122.201  wordpress adminer
```

### Step 6 — Access the application

```
https://wordpress   → WordPress
https://adminer     → Adminer (database manager)
```

## 🔑 Key Variables (`vars/variables.yml.j2`)

| Variable                   | Description                                             |
| -------------------------- | ------------------------------------------------------- |
| `kubernetes_minor_version` | Minor version used in repository URL (e.g. `1.32`)      |
| `kubernetes_version`       | Full package version with suffix (e.g. `1.32.4-1.1`)    |
| `pod_network_ip`           | Pod CIDR for Calico (`10.10.0.0/16`)                    |
| `controller_node_ip`       | Master node IP                                          |
| `kubernetes_operator`      | OS user created for cluster management (`kubernetesop`) |
| `metallb_ip_range`         | MetalLB IP pool (`192.168.122.201-192.168.122.210`)     |
| `proxy_enabled`            | Enable/disable proxy configuration (`false`)            |

## 🧩 Ansible Playbook Structure

```
PLAY 1 — All nodes
  → Base packages, containerd, kubeadm/kubelet/kubectl

PLAY 2 — Master (controller)
  → kubeadm init, Calico CNI, operator kubeconfig

PLAY 3 — Workers (nodes)
  → Join the cluster using the token generated in PLAY 2

PLAY 4 — Master (post-join)
  → local-path-provisioner, ingress-nginx, cert-manager
  (installed after workers join to avoid pods stuck in Pending due to control-plane taint)

PLAY 5 — Master (LoadBalancer)
  → MetalLB, IPAddressPool, L2Advertisement
  → Patch ingress-nginx-controller to LoadBalancer type
```
